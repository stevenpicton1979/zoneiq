# DECISIONS.md

All ambiguity decisions logged here. No questions asked — decided and documented.

---

## D01 — GeoJSON loading strategy: `fs.readFileSync` over static import

**Decision:** Use `fs.readFileSync` at runtime rather than `import zonesGeoJSON from '@/data/brisbane-zones.geojson'`.

**Reason:** The BRIEF shows a static import, but Brisbane's zoning dataset can be 20–80 MB. A static import bundles the file into the serverless function, exceeding Vercel's 50 MB compressed bundle limit and hitting the 512 MB memory limit on the hobby tier. `fs.readFileSync` reads from disk at request time (file is part of the deployment), keeps the bundle small, and avoids any build-time dependency on the file being present during `next build`.

**Trade-off:** Slightly slower cold start on first request (disk read + JSON.parse). Mitigated by a module-level cache so subsequent requests skip disk IO.

## D02 — Placeholder GeoJSON committed to repo

**Decision:** Commit `data/brisbane-zones.geojson` as an empty FeatureCollection `{"type":"FeatureCollection","features":[]}` so the app builds and deploys without the real data.

**Reason:** The real file must be generated by running `npx tsx scripts/download-zones.ts`. Without a placeholder, the app throws at startup. The placeholder allows the app to run and return `OUTSIDE_COVERAGE` for all addresses — acceptable for Sprint 0 setup phase.

**Action required:** After running the download script, replace the placeholder and redeploy.

## D03 — ArcGIS endpoint pagination

**Decision:** The `download-zones.ts` script uses the BCC ArcGIS REST API with pagination (1000 features per page via `resultOffset` and `resultRecordCount`). If the server returns fewer records than requested, pagination stops.

**Reason:** ArcGIS FeatureServer endpoints typically cap responses at 1000–2000 features per request. The Brisbane zoning layer has ~10,000+ features; a single request would be truncated silently.

**Fallback:** If the ArcGIS endpoint fails entirely (HTTP error or zero features after all pages), the script falls back to the BCC Open Data Portal download URL which returns the full dataset in a single request.

## D04 — Zone code property name in BCC GeoJSON

**Decision:** The `zone-lookup.ts` tries the following property names in order: `zone_code`, `ZONE_CODE`, `ZoneCode`, `zone`, `Zone`, `ZONE`. Returns the first non-null match.

**Reason:** BCC data sources use different naming conventions and this is not knowable until the file is downloaded. The BRIEF acknowledges this: "verify actual property name after download". The fallback chain handles ArcGIS (typically UPPERCASE) and Open Data Portal (typically lowercase) formats.

---

## D05 — CORS headers on /api/lookup

**Decision:** Add `Access-Control-Allow-Origin: *` to all responses from `/api/lookup`.

**Reason:** This is a developer API intended for third-party clients (AI agents, PropTech apps). Restricting CORS to same-origin would prevent all external use cases. No auth in Sprint 0, so broad CORS is correct.

---

## D06 — Logging failures are non-blocking

**Decision:** `lookup_log` inserts run as fire-and-forget (no `await`, no try-catch surfaced to caller).

**Reason:** BRIEF explicitly states "non-blocking — don't let logging failure affect the response." A failed log insert must never return a 500 to the API consumer.

---

## D07 — Script runner: `npx tsx`

**Decision:** One-time scripts in `/scripts` are run with `npx tsx <script>`. No ts-node, no compilation step.

**Reason:** `tsx` is a zero-config TypeScript runner that respects `tsconfig.json` paths. It is not in devDependencies (to keep deps lean) but is available via npx. The BRIEF does not specify a runner; `tsx` is the current standard for Next.js projects.

---

## D08 — Service role client only on server

**Decision:** `createServiceClient()` in `lib/supabase.ts` is only ever called from server-side code (API routes, scripts). The anon client is exported as `supabase` for any future client-side use.

**Reason:** `SUPABASE_SERVICE_ROLE_KEY` is not prefixed with `NEXT_PUBLIC_` and must never be exposed to the browser. Separation enforced by naming convention and documentation.

---

## D09 — `format=simple` not implemented in Sprint 0

**Decision:** The `format` query parameter is accepted but `format=simple` returns the same full JSON as `format=json`.

**Reason:** The BRIEF defines `format` as optional with `json | simple` but gives no specification for the simple format shape. Sprint 0 focus is the working API. Simple format can be a Sprint 1 item.

---

## D10 — Nominatim rate limiting not implemented

**Decision:** No rate limiting or retry logic for Nominatim in Sprint 0.

**Reason:** Nominatim's usage policy requires max 1 request/second for non-bulk use. At Sprint 0 scale (minimal users), a single request per API call is fine. Rate limiting should be added before launch at scale. Documented here as a known gap.

---

## D11 — `last_verified` stored as ISO string in seed data, cast to date in DB

**Decision:** `last_verified` in `zone-rules-seed.json` is stored as `"2025-01-01"` (ISO string). Supabase/Postgres accepts this as a valid `date` literal.

---

## D12 — Brisbane boundary check via bounding box

**Decision:** "Outside Brisbane" is determined by checking if the geocoded lat/lng falls outside a bounding box: lat (-28.0 to -27.0), lng (152.6 to 153.6). If no zone polygon matches within this box, return `OUTSIDE_COVERAGE`.

**Reason:** The turf `booleanPointInPolygon` loop naturally returns null for addresses outside all zone polygons. The bounding box pre-check gives a more informative error. The box covers the Brisbane LGA with ~10km margin.
